## 1. Overview

Receives the username/password login request forwarded by the BFF via Feign.
Sequentially queries the member record, verifies the password, and issues a JWT.
The `accessToken` is included in the Response and returned to the BFF, which sets it into the Response Header. It is never directly exposed to the client.
auth-service is **stateless**: its sole responsibility is validating credentials and issuing a signed JWT. It has no awareness of rate limiting or session state ‚Äî all frequency control and user session management are the BFF's responsibility.

For all subsequent APIs that require identity data, auth-service reads from the `X-Member-Id`, `X-Member-Account`, and `X-Member-Role` headers injected by the BFF, via `MemberContextFilter` (auto-configured by `common-biz`). No manual header parsing or Redis access is required. See `spec/common-biz/member-context.md`.

## 3. Service UseCase Flow

**UseCase:** `LoginUseCase.login(LoginRq)`
**UseCase Impl:** `LoginUseCaseImpl`
**Repository:** `MemberJpaRepository` (Spring Data JPA)

> **DataHelper convention:** For complex queries or operations spanning multiple tables, extract a dedicated `*DataHelper` class rather than expanding the service. Simple single-table queries like this one may call `MemberJpaRepository` directly from the service.

| Step | Type | Description |
|------|------|-------------|
| 1 | `[DB READ]` | Query `member` by `account` via `MemberJpaRepository`; not found ‚Üí throw `USER_NOT_FOUND` |
| 2 | `[DOMAIN]` | Check `status == ACTIVE`; if not ‚Üí throw `ACCOUNT_DISABLED` |
| 3 | `[DOMAIN]` | Bcrypt-verify `passwd` against stored `password_hash`; mismatch ‚Üí throw `INVALID_CREDENTIALS` |
| 4 | `[DOMAIN]` | Generate JWT (claims: jti, sub=memberId, userAccount, role, iat, exp) |
| 5 | `[RETURN]` | Return LoginRs (including accessToken for BFF to read via Feign) |

## 4. Database

### MySQL

| Table | Operation | Description |
|-------|-----------|-------------|
| `member` | READ | Query member credentials and Bcrypt hash by `account` |

### Table Schema

#### member

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| member_id | BIGINT | NO | ‚Äî | PK, Snowflake ID, generated by application layer |
| account | VARCHAR(100) | NO | ‚Äî | Login account, Unique |
| password_hash | VARCHAR(255) | NO | ‚Äî | Bcrypt hash |
| name | VARCHAR(100) | NO | ‚Äî | Display name |
| role | ENUM('MEMBER','ADMIN') | NO | 'MEMBER' | User role |
| status | ENUM('ACTIVE','DISABLED') | NO | 'ACTIVE' | Account status |
| created_at | DATETIME | NO | ‚Äî | Record creation timestamp |
| updated_at | DATETIME | NO | ‚Äî | Last update timestamp |

> **Note:** `member_id` is a Snowflake BIGINT with no auto-increment.  
> It is serialized as `String` when returned to the frontend to prevent JavaScript 64-bit integer precision loss.

| HTTP Status | Error Code | Description | Trigger Condition |
|-------------|------------|-------------|-------------------|
| 400 | USER_NOT_FOUND | User does not exist | No record found for `account` |
| 400 | ACCOUNT_DISABLED | Account is disabled | `status != ACTIVE` |
| 400 | INVALID_CREDENTIALS | Password verification failed | Bcrypt mismatch |
| 500 | INTERNAL_ERROR | Unexpected system error | Unhandled exception |
