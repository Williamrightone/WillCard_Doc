# Card Apply — card-service Spec

> **Service:** card-service
> **Contract Module:** card-contract
> **Caller:** bff → [mobile-bff/card-apply.md](../mobile-bff/card-apply.md)
> **Last Updated:** 2026-03

---

## 1. Overview

Receives the card application request from BFF via Feign.
Validates that the user does not already hold an active card of the same type.
Generates a mock PAN (BIN prefix + `SecureRandom` digits + Luhn validation), derives `pan_masked` and `pan_hash`, generates a one-time CVV, computes `cvv_itoken` (HMAC-SHA256 of the CVV — stored in DB; raw CVV is discarded after this step), calculates the expiry date (application date + 5 years), encrypts all sensitive fields with AES-256-GCM, and persists the card record with `status = ACTIVE`.
Returns the card details including the **one-time CVV** and **cvvItoken** to BFF.

> **CVV:** Generated by `SecureRandom`. Never stored in DB, Redis, or logs. Exists only in JVM heap during this request and is included once in the response for the user to see.

> **cvv_itoken:** HMAC-SHA256 of the raw CVV using `CVV_HMAC_KEY` (from env/KMS). Stored in DB. Used by Phase 8 (`auth/authorize`) to verify the CVV iToken passed by the App without re-computing HMAC at auth time and without ever storing the raw CVV (PCI DSS compliance).

> **pan_hash:** HMAC-SHA256 of the full PAN. Stored with a unique index. Required by Phase 8 (Issuer Authorization) for PAN-to-member lookup without storing plain PAN.

## 3. Service UseCase Flow

**UseCase:** `CardApplyUseCase.apply(CardApplyRq)` (card-service)
**UseCase Impl:** `CardApplyUseCaseImpl`
**Repository:** `CardJpaRepository` (Spring Data JPA)

| Step | Type | Description | Domain Service |
|------|------|-------------|----------------|
| 1 | `[DB READ]` | Query `card` WHERE `user_id = X-Member-Id` AND `card_type = cardType` AND `status != 'CANCELLED'`; row found → throw `CARD_ALREADY_EXISTS` (CA00009) | `CardQueryService.findActiveByUserAndType()` |
| 2 | `[DOMAIN]` | Generate 16-digit PAN: select BIN prefix by cardNetwork (VISA: `4`, MC: `51`, JCB: `35`), fill remaining digits with `SecureRandom`, validate with Luhn algorithm | `CardNumberService.generate(cardNetwork)` |
| 3 | `[DOMAIN]` | Generate `pan_masked`: replace first 12 digits with `****-****-****`, keep last 4 | `CardNumberService.mask(pan)` |
| 4 | `[DOMAIN]` | Generate `pan_hash`: `HMAC-SHA256(pan, PAN_HMAC_KEY)` | `CardEncryptionService.hashPan(pan)` |
| 5 | `[DOMAIN]` | Generate CVV: `SecureRandom` 3-digit string | `CardNumberService.generateCvv()` |
| 6 | `[DOMAIN]` | Compute CVV iToken: `cvv_itoken = HMAC-SHA256(cvv, CVV_HMAC_KEY)`; zero-fill the intermediate CVV byte buffer after computation; `CVV_HMAC_KEY` injected from env/KMS | `CardEncryptionService.hashCvv(cvv)` |
| 7 | `[DOMAIN]` | Calculate expiry: `applicationDate + 5 years`, format as `MMYY` for storage, `MM/YY` for response | `CardExpiryService.calculate()` |
| 8 | `[DOMAIN]` | AES-256-GCM encrypt: `pan` → `pan_encrypted`; `cardholderNameEn` → `cardholder_name_en`; `cardholderNameZh` → `cardholder_name_zh`; expiry `MMYY` → `expiry_date` | `CardEncryptionService.encrypt()` |
| 9 | `[DB WRITE]` | Insert `card` record: `card_id` = Snowflake, `user_id` = X-Member-Id, `card_type`, `card_network`, `status = ACTIVE`, `pan_encrypted`, `pan_masked`, `pan_hash`, `cvv_itoken`, `cardholder_name_zh`, `cardholder_name_en`, `expiry_date`, `activated_at = now()` | `CardJpaRepository.save()` |
| 10 | `[RETURN]` | Return `CardApplyRs` (cardId, maskedPan, cardType, cardNetwork, expiryDate in `MM/YY`, cardholderNameEn decrypted, cvv, status = `ACTIVE`, activatedAt) — **raw CVV is passed from Step 5 in JVM heap only, never persisted** | — |

### Table Schema

#### card

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| card_id | BIGINT | NO | — | PK, Snowflake ID (application-generated) |
| user_id | BIGINT | NO | — | Member ID; referential integrity enforced at application layer |
| card_type | VARCHAR(20) | NO | — | CLASSIC / OVERSEAS / PREMIUM / INFINITE |
| card_network | VARCHAR(10) | NO | — | VISA / MASTERCARD / JCB |
| status | VARCHAR(20) | NO | — | PENDING / ACTIVE / FROZEN / CANCELLED |
| pan_encrypted | VARCHAR(512) | NO | — | AES-256-GCM encrypted full PAN |
| pan_masked | VARCHAR(20) | NO | — | `****-****-****-1234`; stored as plain text for display |
| pan_hash | VARCHAR(64) | NO | — | HMAC-SHA256 of PAN; used for Phase 8 PAN-based lookup; **Unique Index** |
| cvv_itoken | VARCHAR(64) | NO | — | HMAC-SHA256 of CVV using `CVV_HMAC_KEY`; used for Phase 8 CVV verification; raw CVV never stored |
| cardholder_name_zh | VARCHAR(512) | NO | — | AES-256-GCM encrypted Chinese name |
| cardholder_name_en | VARCHAR(512) | NO | — | AES-256-GCM encrypted English name |
| expiry_date | VARCHAR(512) | NO | — | AES-256-GCM encrypted MMYY |
| activated_at | DATETIME(3) | YES | NULL | Timestamp of first ACTIVE transition; written once, never overwritten |
| cancelled_at | DATETIME(3) | YES | NULL | Timestamp of CANCELLED transition; written once, never overwritten |
| created_at | DATETIME(3) | NO | — | Inherited from BaseTimeEntity |
| updated_at | DATETIME(3) | NO | — | Inherited from BaseTimeEntity |

> **Application-layer constraint:** `(user_id, card_type)` WHERE `status != 'CANCELLED'` — enforced by Step 1 duplicate check, not a DB-level partial unique index.
> **Unique Index:** `pan_hash` — prevents duplicate PAN across all users.
> **No unique index on cvv_itoken:** 3-digit CVV has only 1,000 possible values; uniqueness is not meaningful. Lookup is always by `card_id`.

## 6. Changelog

### v1.0 — 2026-03 — Initial spec

### v2.0 — 2026-03 — CVV iToken support

**Background:** PCI DSS prohibits storing raw CVV (SAD). A one-way HMAC-SHA256 token (`cvv_itoken`) is computed at card issuance and stored in DB. At Phase 8 authorization, Mock NCCC retrieves `cvvItoken` from the internal card-plain endpoint and includes it in `encryptedCard`; card-service decrypts and compares the received token against the stored value — no raw CVV is ever persisted or re-transmitted after issuance.

**Section 1 — Overview**
- Added description of `cvv_itoken` computation and its role in Phase 8 authorization.
- Clarified that raw CVV exists in JVM heap only during this request; `cvv_itoken` is what gets persisted.

**Section 2 — Response (`CardApplyRs`)**
- `cvv` field: notes updated to "One-time display only; never stored by App".
- `cvvItoken` field: **not added to response** — the iToken is an internal server-side value and is never exposed to the client.

**Section 3 — UseCase Flow**
- **Step 6 (new):** `CardEncryptionService.hashCvv(cvv)` — compute `cvv_itoken = HMAC-SHA256(cvv, CVV_HMAC_KEY)`; zero-fill CVV byte buffer after computation; `CVV_HMAC_KEY` is injected from env/KMS.
- **Steps 6–9 (renumbered to 7–10):** original Step 6 (expiry) → Step 7; Step 7 (encrypt) → Step 8; Step 8 (DB write) → Step 9; Step 9 (return) → Step 10.
- **Step 9 (was Step 8) — DB WRITE:** added `cvv_itoken` to the INSERT column list.
- **Step 10 (was Step 9) — RETURN:** no change to response fields; note updated to reflect that `cvv_itoken` is persisted (not returned).

**Section 4 — Table Schema (`card`)**
- Added column `cvv_itoken VARCHAR(64) NOT NULL` positioned after `pan_hash`.
- Added note: no unique index on `cvv_itoken` (3-digit CVV has only 1,000 possible values; lookup is always by `card_id`).
- **New Flyway migration required:** `V1.0.5__add_card_cvv_itoken.sql`
  ```sql
  ALTER TABLE `card`
    ADD COLUMN `cvv_itoken` VARCHAR(64) NOT NULL
      COMMENT 'HMAC-SHA256 of CVV; CVV_HMAC_KEY from env/KMS; never stores raw CVV (PCI DSS)'
    AFTER `pan_hash`;
  ```
